01 · THE PROBLEM

Why streaming
is hard

Delivering video to millions of simultaneous viewers across wildly different devices and network conditions requires something smarter than a file download.

01

Bandwidth is unpredictable

A viewer's connection can swing from 50 Mbps to 2 Mbps in seconds — commutes, crowded stadiums, bad WiFi. A fixed-quality file either buffers or plays in low resolution forever.

02

Devices are everywhere

The same content needs to work on a 4K TV, a phone on LTE, a laptop on hotel WiFi, and a Roku stick — each with different hardware capabilities and network constraints.

03

Scale is massive

A popular live event can spike to millions of concurrent viewers in seconds. A single origin server can't handle that — content must be distributed globally before viewers arrive.

02 · THE SOLUTION

Adaptive bitrate
streaming

Instead of one large video file, the stream is pre-encoded at multiple quality levels and sliced into small segments. The player picks the right quality in real time — switching silently as conditions change.

SOURCE
Raw video input
A camera feed, file, or live encoder pushes raw video — often ProRes or H.264 at a very high bitrate — into the encoding pipeline.
INPUT
ENCODE
Multi-bitrate transcoding
AWS MediaConvert transcodes the source into multiple renditions simultaneously — 1080p at 5 Mbps, 720p at 2.5 Mbps, 360p at 800 Kbps — each a complete independent stream.
MEDIACONVERT
SLICE
Segmentation into chunks
Each rendition is split into short segments — typically 6 seconds. Each segment is independently decodable, letting the player switch quality at any segment boundary without buffering.
~6s SEGMENTS
MANIFEST
Playlist / manifest generation
A manifest file is generated describing all renditions and segment locations. HLS uses .m3u8 playlists. DASH uses XML-based .mpd files. The player reads this first to understand what's available.
.m3u8 / .mpd
ORIGIN
S3 origin storage
Segments and manifests are stored in an S3 bucket — the authoritative origin. This project uses a real S3 bucket with static website hosting as its origin server.
AWS S3
CDN
CloudFront edge caching
CloudFront distributes segments to 400+ edge locations globally. Players fetch from the nearest edge node — often milliseconds away. The origin only serves each segment once per region.
CLOUDFRONT
PLAYER
HLS.js ABR player
HLS.js runs an ABR algorithm in the browser — monitoring bandwidth and buffer health every few seconds, selecting which rendition to fetch next. You can watch this live in the Demo.
HLS.JS
03 · THE PROTOCOLS

Three ways ABR
is delivered

The streaming industry settled on three interlocking standards. Understanding how they relate is core to understanding platforms like these.

PROTOCOL_01
HLS
HTTP Live Streaming

Apple's protocol, required on iOS and Safari. Uses .m3u8 playlists. The most widely deployed streaming protocol on the planet.

EXPLORE HLS →
PROTOCOL_02
DASH
Dynamic Adaptive Streaming over HTTP

The open ISO standard. Codec-agnostic, preferred on Android and Chrome. Uses XML-based MPD manifests. Powers YouTube.

EXPLORE DASH →
PROTOCOL_03
CMAF
Common Media Application Format

The unifier. One set of fMP4 segments served to both HLS and DASH players. Encode once, reach every device. The industry direction.

EXPLORE CMAF →
